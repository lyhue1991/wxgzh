# SKILL 小白入门：从知道是什么，到自己动手做一个

这两年，大家都在聊 Agent、MCP、工作流自动化。

但很多人第一次接触 `SKILL` 时，都会有一个很真实的问题：它到底是什么？和我平时写的 Prompt 有什么区别？又和 MCP 有什么关系？

这篇文章就尽量不用太绕的术语，把 `SKILL` 讲清楚。

如果你是第一次接触 Agent 生态，看完这篇，至少能搞明白四件事：

1. `SKILL` 到底解决什么问题
2. 它和传统 Prompt、MCP 分别负责什么
3. 去哪里找、怎么安装
4. 怎么自己动手做一个属于你的 skill

## 一、SKILL 基本概念

先说一句最容易理解的话：

`SKILL` 可以理解成给 Agent 安装的“专项能力包”。

它不是一句临时 Prompt，也不是一个外部工具服务，而是一套可以被反复调用的操作经验、工作流程和领域知识。

比如：

- 你想让 Agent 写出来的前端页面更有设计感，而不是千篇一律，那可以装 `frontend-design`
- 你要处理 PDF，提取文字、表格、元数据，可以装 `pdf`
- 你要和 Agent 一起系统化地写文档，可以装 `doc-coauthoring`
- 你甚至想自己“造 skill”，可以用 `skill-creator`

### 1）SKILL 的核心作用

一个 skill 本质上是在告诉 Agent：

- 遇到什么类型的问题时，应该启用我
- 启用之后，应该按什么流程思考和执行
- 需要时，再去读取哪些额外资源、脚本、参考资料

所以它的价值，不只是“多了一段提示词”，而是把一类任务沉淀成了可复用的方法论。

### 2）什么叫“渐进披露”

这是 skill 很重要、也很容易被忽略的一点。

skill 不是一上来就把所有内容都塞进上下文里，而是分层加载：

1. **元数据层**：先让模型知道 skill 叫什么、适合什么场景
2. **主体说明层**：当 skill 被触发后，再把 `SKILL.md` 的主体说明放进上下文
3. **资源层**：如果还需要脚本、参考文档、模板，再按需读取

这就是“渐进披露”的意思：**需要多少，加载多少**。

它带来的好处很直接：

- 不会像超长 Prompt 一样，一开始就把上下文塞爆
- skill 多了以后，Agent 也更容易按需调用，而不是每次都全量背书
- 复杂能力可以拆层组织，不容易乱

你可以把它想象成一本结构清楚的工具书：先看目录，再翻章节，最后才查附录，而不是把整本书一次性背下来。

### 3）和传统 Prompt 的区别

很多人会问：既然我都能写 Prompt，为什么还要 skill？

因为二者解决的问题层级不一样。

**Prompt** 更像一次性的口头交代。

比如你说：

> 帮我写一个有设计感的 landing page，不要太像 AI 模板。

这当然也能做，但每次都要重新说，而且说得是否完整、是否稳定，结果波动会很大。

**SKILL** 更像把这套经验正式封装起来。

它会明确告诉 Agent：

- 什么时候该触发
- 什么样的审美方向值得优先考虑
- 怎么避免“默认 AI 味”
- 需要时用哪些资源和步骤完成任务

所以一句话概括：

- `Prompt` 解决“这次你帮我做什么”
- `SKILL` 解决“这类事以后都该怎么做”

### 4）和 MCP 的区别

如果说 skill 是“经验包”，那 MCP 更像“工具接口”。

MCP（Model Context Protocol）解决的是：**Agent 怎么连接外部世界**。

比如：

- 连 GitHub
- 连数据库
- 连本地文件系统
- 连浏览器、飞书、Slack、Notion 等服务

MCP 的强项是“拿数据、调工具、做动作”。

而 skill 的强项是“告诉 Agent 遇到某类任务时，应该怎么做得更专业”。

所以它们不是替代关系，而是非常适合配合：

- `SKILL` 决定做事的方法
- `MCP` 提供做事的手和脚

### 5）三者优缺点对比

| 方式 | 它最像什么 | 优点 | 缺点 | 适合场景 |
| --- | --- | --- | --- | --- |
| Prompt | 临时交代 | 上手最快、零门槛 | 不稳定、不可复用、容易遗忘细节 | 一次性任务、快速试验 |
| SKILL | 可复用经验包 | 结构化、可复用、可渐进加载、适合沉淀流程 | 需要设计触发条件和结构 | 某类任务经常做，希望稳定输出 |
| MCP | 外部工具接口 | 能访问真实数据和服务，执行能力强 | 配置更复杂，涉及权限和安全 | 需要连接 GitHub、数据库、浏览器等外部系统 |

如果你是小白，可以先这样理解：

- 先用 Prompt 跑起来
- 某件事反复要做，就把它沉淀成 skill
- 需要连外部系统时，再接入 MCP

这个学习路径通常最顺。

## 二、SKILL 查找与安装

### 1）去哪里找 skill

目前比较值得看的一个站点是：

- [skills.sh](https://skills.sh/)

它可以理解成一个公开的 skill 目录站。你可以在里面：

- 看热门 skill 排行
- 按关键词搜索 skill
- 点进 skill 详情页看说明、仓库、安装命令
- 看哪些 Agent 支持这类 skill

如果你是新手，最简单的做法就是先去 `skills.sh` 搜关键词：

- 想做前端，就搜 `frontend`
- 想写文档，就搜 `doc`
- 想处理 PDF，就搜 `pdf`
- 想自己做 skill，就搜 `skill-creator`

另外，你也可以参考 OpenClaw 生态常见的 skill 站点：

- [clawhub.ai](https://clawhub.ai/)

可以把它当作另一个逛 skill 生态、找灵感、看社区沉淀的入口。很多时候，一个站点没想到的关键词，去另一个站点看看，反而更容易发现宝藏。

### 2）怎么安装 skill

`skills.sh` 的文档里给出的基础命令是：

```bash
npx skills add <owner>/<repo>
```

比如：

```bash
npx skills add vercel-labs/agent-skills
```

而在具体 skill 的详情页中，你通常还能看到可直接复制的安装命令。很多 skill 支持直接给 GitHub 仓库地址，再配合 `--skill` 指定具体 skill，例如：

```bash
npx skills add https://github.com/anthropics/skills --skill frontend-design
```

这也是很适合新手的方式：**看到哪个 skill 喜欢，就直接复制详情页的命令安装。**

### 3）安装时你可以怎么理解

你不需要把安装 skill 想得太重。

它本质上不是在装一个重量级应用，更像是在把一套可触发的能力描述安装到 Agent 的 skill 目录里。之后 Agent 在合适的时候，就能识别并调用它。

所以你可以把安装过程理解成：

> 给你的 Agent 多装一个“遇到这类任务时的专业打法”。

## 三、几个常用 SKILL 推荐

如果你刚开始接触，我建议不要一上来就装几十个。先挑 3 到 5 个你马上能用上的，体感会更明显。

### 1）frontend-design

这是一个非常适合前端开发者体验“skill 和普通 Prompt 差异”的 skill。

它的目标非常明确：**生成有设计感的前端代码，而不是默认 AI 味很重的页面。**

它会强调很多具体原则，比如：

- 先明确视觉方向，而不是直接堆组件
- 避开过度常见的默认字体和配色
- 在排版、颜色、动效、背景层次上做出风格判断
- 让页面有记忆点，而不是只求“能跑”

如果你平时已经受够了“白底紫渐变、圆角卡片、Inter 字体”的 AI 套路感，这个 skill 很值得装。

安装示例：

```bash
npx skills add https://github.com/anthropics/skills --skill frontend-design
```

### 2）frontend-slides

这类 skill 的思路很实用：**不用传统 PPT，而是直接用 HTML/CSS/JS 生成 slides。**

它的好处在于：

- 演示稿可版本管理
- 能直接复用前端能力做动效和排版
- 更容易做出网页级演示，而不是模板化 PPT
- 适合技术分享、产品 demo、发布会样稿

如果你本来就会一点前端，那么“用代码写 slides”会比手动拖 PPT 舒服很多。

你可以在 [skills.sh](https://skills.sh/) 或 [clawhub.ai](https://clawhub.ai/) 搜 `frontend-slides` 这一类关键词，通常都能找到对应思路或同类 skill。

### 3）doc-coauthoring

这是一个特别适合写文档的 skill。

它不是简单地“帮你写文章”，而是把写文档这件事拆成三阶段协作：

1. **背景收集**：先把上下文补齐
2. **结构与细化**：逐段讨论、逐段成文
3. **读者测试**：用更像“陌生读者”的视角检查文档是否真的讲清楚了

这个 workflow 很适合：

- 技术方案
- PRD
- RFC
- 决策文档
- 团队内部说明文

它最大的价值在于：让 Agent 不只是“代笔”，而是“共创”。

安装示例：

```bash
npx skills add https://github.com/anthropics/skills --skill doc-coauthoring
```

### 4）pdf

这个 skill 非常接地气。

现实工作里，PDF 是高频格式，但也很烦：抽文字、抽表格、合并拆分、识别扫描件，都很容易卡住。

`pdf` skill 就是专门把这类事情流程化，常见用途包括：

- 提取 PDF 文本
- 提取表格
- 合并、拆分、旋转 PDF
- 读取元数据
- 处理扫描版 PDF 的 OCR 场景

如果你的工作经常跟论文、报告、合同、材料打交道，这个 skill 非常实用。

安装示例：

```bash
npx skills add https://github.com/anthropics/skills --skill pdf
```

## 四、SKILL 动手构建

会用 skill，只是第一步。

真正更有意思的是：**你可以把自己反复使用的一套流程，也做成一个 skill。**

比如：

- 固定风格的公众号写作流程
- 某类数据分析报告生成流程
- 你团队内部的代码审查规范
- 某种特定技术栈的脚手架流程

当你把这些经验固化下来，Agent 的输出就会更稳定，而且团队也能复用。

### 1）SKILL 的结构规范

一个典型 skill 的结构并不复杂，通常长这样：

```text
my-skill/
├── SKILL.md
├── scripts/
├── references/
└── assets/
```

其中最核心的是 `SKILL.md`。

它通常包含两部分：

1. **frontmatter 元信息**：例如 `name`、`description`
2. **正文说明**：告诉 Agent 这个 skill 什么时候触发、触发后该怎么做

而其他目录可以按需补充：

- `scripts/`：放可重复执行的脚本
- `references/`：放较长的参考文档
- `assets/`：放模板、素材、资源文件

初学者最需要记住的一点是：

**一个好 skill，不是把内容写得越长越好，而是要让 Agent 清楚知道“什么时候该用我，以及用了以后怎么走”。**

### 2）使用 skill-creator 构建

如果说别的 skill 是专项技能，那 `skill-creator` 很像“skill 之母”。

它是专门用来帮助你创建、测试、迭代、优化 skill 的。

你可以把它理解成一个“造 skill 的 workflow 教练”，它会引导你去想清楚：

- 我这个 skill 到底要解决什么问题
- 什么时候应该触发
- 输出应该长什么样
- 怎么设计测试样例
- 怎么比较迭代前后的效果
- 怎么继续优化 skill 描述，让触发更准确

它特别适合下面几类人：

- 已经有一套固定工作流，想沉淀出来
- 已经会写 Prompt，但希望升级成可复用能力包
- 想给自己的 Agent 或 OpenClaw/OpenCode 一类环境补充更专业的能力

安装示例：

```bash
npx skills add https://github.com/anthropics/skills --skill skill-creator
```

对于新手来说，我建议这样入门：

1. 先观察自己有哪些重复任务
2. 挑一个最常见、边界最清晰的任务
3. 用 `skill-creator` 写出第一版 skill
4. 跑几个真实例子，看看哪里不稳
5. 再继续迭代，而不是一开始就追求“大而全”

## 最后一句

如果你以前只把 Agent 当成“高级聊天机器人”，那 skill 会让你开始真正理解：**Agent 的能力，不只是模型本身，还来自它背后的可复用工作流。**

Prompt 让你把一次需求说出来。

Skill 让你把一类经验沉淀下来。

MCP 让 Agent 真正连上外部世界。

这三者一旦配合起来，Agent 才会越来越像一个真正能干活的数字同事。

如果你想开始动手，最简单的路径就是：

- 先去 [skills.sh](https://skills.sh/) 逛一圈
- 再去 [clawhub.ai](https://clawhub.ai/) 看看社区生态
- 先装一个 `frontend-design` 或 `pdf`
- 最后再挑战一下 `skill-creator`

你会很快发现：给 Agent 装 skill，和给自己装新能力，几乎是同一件事。
